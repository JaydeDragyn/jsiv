Plan for the UserManual window

This will be a JFrame similar to the main ProgramWindow, but will only provide
access to the user manual, a links panel, and a Navigation menu.

Navigation menu will have
    Hide Navigation     - Collapse/hide the Section Links panel
    Show Navigation     - Expand/show the Section Links panel
    Previous Link       - Show the previous link in right window
    Next Link           - Navigate forward a link
    Close User Manual   - Close this window

Left panel will be a collapsible JPane.  Collapsible so the user can have the
right panel expand and show more information.
This panel will display section links gleaned from the html that is displayed
in the right panel.  Clicking on one of the section links will be the same
as clicking on the section link in the table of contents, scrolling the
right window to the relevant section.

Right panel will be an html display that will basically display the content
as a web page.  Nothing super fancy - some CSS, enough for headers, link
navigation, tables, and images (including animated gifs).

The left panel will shrink down and the button will change to a right-pointing
arrow.  When clicked, the panel will expand and button will change back to be
the first image above.

This will keep an empty panel on the left side of the window, but it will
ensure the user knows there is a panel there that they can expand for more
navigation options, and it should be thin enough to not take up much room
so the content panel can expand meaningfully (unless the user shrinks the
window down, but that's the user's fault).

-------------

Left pane can be two internal panes, side-by-side; left pane the content pane (containing
the section links), right pane a single button that fills the entire pane and
simply has either a > or < as the button label (maybe use an actual filled in
<| or |> character if we can find a unicode character that exists in the
default font set in both windows and unix environments, otherwise just use
the < and > characters).

By default, both sides of the left pane will be visible.  Pressing the <|
button will collapse this left pane so only the |> button is visible.
Pressing this |> button will expand the pane so both sides are visible again.

Reasoning for this decision: The expand/collapse button will always be visible
and will always take up the exact same amount of space.  The button filling
the entire vertical space of the right pane means the user can click on it
anywhere to activate the expand/collapse behavior, and even if the window is
resized, the button will take up the same relative amount of space, so it
should never become impossible for the user to see it and interact with it.


--- Behavior ---

Upon opening, both the Left and Right sections will be visible, so the user
will see a Navigation Links section on the left that contains section
header links and a collapse button, and the beginning of the User Manual html
document on the right.

Scrolling in either section of text will scroll that section up or down like
any standard document, web page, etc.

Ctrl-Scrolling in either section will reduce or enlarge the text (within
minimum and maximum size limits) to make it easier to read or show more
information based on user preference.

Clicking on a link in either section will cause the right section to scroll
such that the associated section header moves to the top of that right
section, "navigating to that section".

Clicking on the Collapse button or Navigation->Hide Navigation will hide the
Section Navigation section and change the button to an Expand button (with a
|> centered vertically on the button), leaving only the Expand button visible
on the left side.  This will shift over to the left so the right section is
wider/takes up more horizontal space so more text is visible.

Clicking on the Expand button or Navigation->Show Navigation will show the
Section Navigation section and change the button to a Collapse button (with
a <| centered vertically on the button).  The left edge of the right section
will shift over to the right so less text is visible and the right section
takes up less horizontal space, but the navigation links text will be visible
on the left with the Collapse button between the two sections.  The Collapse
button will have no space between it and the right edge of the Section
Navigation text area, but will have the same amount of space as the window
borders between the right edge of the button and the left edge of the right
section.

Navigation->Previous or Left Arrow will scroll the right section so the
last navigated-to section header is at the top of that right section,
"navigating to the previously chosen section".  If we are at the beginning
of the list (or no section navigations have occurred) then this will cause
the right section to scroll to the top, effectively returning the user to
the "beginning" of the User manual.

Navigation-Next or Right Arrow will scroll the right section to the next
navigated-to section in the navigation list, that section title moving to
the top of the right section, "navigating to the next chosen section".

As the user navigates to sections by clicking on section headers (whether
they click on the left or right side), the UserManualWindow will keep a
list of those navigations - up to 50 entries.  There are only 26 sections
in this User Manual, so this will give the user the ability to navigate
to several sections in a specific order of their choosing, and then
quickly navigate among them with the left/right arrow keys.  Having the
list at 50 will allow the user to place almost every section into that
list twice, with the assumption that they probably won't need to put every
section in the list twice, and most likely will only really jump around
between a handfull of sections anyway.

Navigation->Close User Manual or pressing the X on the UserManualWindow will
hide the UserManualWindow (setVisible(false);).  This window can be brought
back by pressing F1 or Help->User Manual on the main ProgramWindow, which
will simply setVisible(true); so the user manual returns with the previous
state (navigation list, Section Navigation section shown or hidden as it was
when the UserManualWindow was hidden, right section scrolled to the same
place as when it was hidden, etc.).

Original idea for link tracking was going to be to keep track of the most
recent 20 or so links the user clicks on and enable them to go forward and
backward through that list.  While useful, it isn't very intuitive, which
is something I've been musing over but only tonight thought of a better way
to implement it that will be intuitive.

Instead of a hidden list of the links, have a Bookmarks menu and a Help
menu, and change the Navigation menu to a View menu.

View
  +- Toggle Navigation Pane
  +- ---
  +- Exit User Manual (Ctrl-E)

Bookmarks
  +- Add current link as Bookmark (Ctrl-LMB)
  +- Remove current link from Bookmark (Ctrl-Shift-LMB)
  +- ---
  +- Next Bookmark (Ctrl - ->)
  +- Previous Bookmar (Ctrl - <-)
  +- ---
  +- <Bookmark 1>
  +- <Bookmark 2>
     ...
  +- <Bookmark n>

Help
  +- Using User Manual Viewer
  +- About User Manual Viewer

So we will still keep track of up to n links, but as bookmarks that the user
will explicitly add.  This will allow the user to put up to n sections in the
bookmarks menu to go to quickly instead of having to scroll around in the
document to find the index and locate the section they want to view again.

The Help menu will be far simpler - both options will open a modal dialog
to display information with a single [Ok] button to close them.

Help->Using User Manual Viewer will show something like:

  This viewer displays the <JSIV User Manual>.
  On the left is a section navigator and on the right is the content.
  You can manually scroll the content, you can click on links in the
    section navigator, and you can click on links in the content to
    go to other parts of the content.

  If you Ctrl-Click a link, it will save as a bookmark, accessible
    in the Bookmarks menu.  Using Ctrl and the left or right arrow keys
    will go through the bookmarks you have added.

  The section navigator can be hidden by clicking on the button with
    the < symbols, and can be restored by clicking on the button with
    the > symbols.

Note that <JSIV User Manual> will be whatever the client program set as the
title with the UserManualWindow.setTitle() method.  If nothing is set, then
this will default to "This viewer displays the user manual for the program
you are using".

Help->About User Manual Viewer will show something like:

For the View menu:
* I might add Enlarge Text, Reduce text and Reset text size to
  enlarge/shrink the text in the content pane if I can figure out how
  to do this easily.
  These would use Ctrl-'+', Ctrl-'-' and Ctrl-0 respectively

* Neither the Toggle nor the Close options will have Accelerator keys.
  - I don't want to interfere with Ctrl-C (system-wide Copy), using a
    different Accelerator key than 'C' for 'Close' is awkward, and I
    want to use "Close" instead of "Exit" since this will close the
    User Manual Viewer, not actually exit or terminate anything.
  - The Toggle option most likely won't be used much, so requiring Menu
    access is fine.

For the Help Menu:
* Neither option will have an Accelerator key.
  - The "Using" option will bring up a modal dialog that shows a brief
    blurb that explains the sections of the User Manual Window.
  - The About option will bring up a modal dialog that provides info
    about this User Manual Window component, separate from whatever
    program it is showing a User Manual for.

For the Navigation menu:
* These will just be the section headers the user last clicked on (it will
  take the link text and use that).
  - When the user clicks on a link, if it was the same link that was last
    clicked on (and thus in the list at the top) then we will not record
    a second copy of that link in a row.
* Determining whether we will include Back and Forward options to navigate
  through the list of links:
  - Pro: If the user accidentally clicks on a link and wants to go back,
      having a back option will be useful.  This mirrors behavior in
      web browsers.
  - Pro: If the user lines up a couple of links strategically, they can
      use the Back and Forward options will be useful.
  - ??: If the user goes back in the list of links, even if we show them
      where they are in the list (put a checkmark next to the Navigation
      menu option that is where they most recently clicked), how do we
      handle them clicking on a new link?
      1) Put the new link at the end of the list and jump to that link.
        But going "Back" would not necessarily go back to the link they
        might expect.
      2) Clear all links "after" the last clicked link, and then put the
        newly clicked link at the end of the list.  This mirrors web
        browser behavior.
  - Final decision:
    - Include a Back option
    - Treat the navigation link history like a stack that only holds 10
      items.  An 11th item will clear the oldest item so there are still
      only 10.
    - Going back to link X and then clicking on a new link will erase all
      links above link X in the stack, and then put the new link on top.
    - No forward option - that will be clicking on a new link (or clicking
      on one of the links "above" the current position in the Navigation
      menu).
    * Rationale: Going "back" to a link that the user clicked on previously
      is intuitive and has a clear place to go.  But if the user scrolls
      to one of the "future" links and then uses the "forward" option, it
      is not intuitive where they will end up.  Having the last 10 links
      they clicked on in the Navigation menu will give them the ability
      to do that, and clicking on a new link will just add to the stack
      as usual so the list in the Navigation menu is truly just a list
      of the last 10 links they clicked on.
      Exception: If the user clicks on one of the links in the Navigation
      menu (whether using the menu, using a link in the Content pane or
      a link in the Section Navigation pane), we'll move the Content
      pane to that link, and set our internal link index to that link's
      index instead of erasing other links.  Only if they click on a
      link that is not already in the last 10 will we remove/clear the
      links above the current link.

Final behavior:

Navigation stack with up to 10 of the most recently clicked links:
  - Clicking on a link that is already in the list (whether the user
    clicks on the link in the list, on the content pane or in the
    section navigation pane) will move that link to the top of the stack.
  - The stack will have a position index.
    - This position index will normally point to the most recent link
      (top of the stack).
    - If the user clicks on a link, this position index will change to
      point to the top of the stack after the link is placed at the top
      of the stack.
    - If the user uses the Back option, this position index will change
      to point to the next item down the stack, and the menu will show
      an icon next to the link and then navigate the content pane to
      that link.
      - If the user goes all the way back to the oldest link and tries
        to go "Back" again, the content pane will scroll back to the
        top of the document.
  - Clicking on a link (in the content pane or section navigation pane)
    that is not already in the list will push that link to the top of
    the stack, return the position pointer to that top-of-stack link,
    and then navigate the content pane to the clicked link.
